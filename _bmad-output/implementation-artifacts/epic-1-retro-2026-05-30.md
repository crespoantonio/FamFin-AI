## 2. Retrospective Discussion Summary

### What Went Well
* **Pivot to n8n Webhook Integration:** Shifting the direct Telegram integration out of the FastAPI application and into n8n simplified our API routing layer significantly. It allows us to manage message channels externally while keeping the Python backend secure and focused on business logic.
* **Robust Application Encryption:** Implementing `cryptography.fernet` (AES-128 in CBC with HMAC-SHA256) inside [encryption.py](file:///c:/Users/cresp/Documents/Projectos/FamFin-AI/src/core/encryption.py) provides authentic encryption out of the box and prevents padding oracle attacks.
* **Multi-Tenant Scoping:** The `User` and `Family` tables were structured with globally unique UUIDv4 primary keys, and relationships were setup with cascading deletes for GDPR compliance.
* **Test Coverage:** Establishing 15 passing tests early on ensured that we have high confidence in our security, registration, and API routing.

### Challenges and Growth Areas
* **Error and Input Resilience:** The encryption service initially crashed on missing or invalid keys. We successfully patched these to raise controlled exceptions and avoid application downtime.
* **Security Hardening:** Initial direct string comparisons on secret headers were replaced with constant-time comparison `secrets.compare_digest` to prevent timing attacks.
* **Database Migration Plan:** We currently run SQLModel table creation on application startup. As our models evolve in future epics, this will become unsustainable without a migration tool like Alembic.
* **Containerization File Permissions:** Mounting directories from a Windows host into rootless containers can cause permission mismatches. We worked around this by testing against in-memory databases, but a permanent compose fix is needed.
